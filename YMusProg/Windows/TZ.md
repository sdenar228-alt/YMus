# Техническое задание: YMus Desktop для Windows

## 1. Общая информация

**Название продукта:** YMus Desktop  
**Первая целевая платформа:** Windows  
**Последующая платформа:** macOS  
**Тип продукта:** графическое desktop-приложение для установки и обновления
браузерного расширения YMus.

### 1.1. Назначение

YMus Desktop должен упростить пользователю установку актуальной версии
расширения YMus в один или несколько браузеров на компьютере.

Программа должна:

- находить поддерживаемые браузеры;
- показывать их в понятном списке;
- определять статус установки YMus;
- получать с сервера информацию о последней версии расширения;
- скачивать подходящие пакеты;
- проверять целостность и подлинность пакетов;
- устанавливать или подготавливать расширение к установке выбранным браузерам;
- обновлять ранее установленную локальную версию;
- объяснять действия, которые пользователь обязан подтвердить в браузере.

### 1.2. Основной принцип

Программа не должна обходить защитные механизмы браузеров. Для каждого браузера
используется только поддерживаемый им способ установки.

На обычном домашнем компьютере полностью тихая установка произвольного
расширения недоступна во всех браузерах:

- Chrome на Windows не разрешает локальную установку CRX обычным приложением;
- Chrome и Edge ограничивают внешнюю принудительную установку корпоративными
  и магазинными сценариями;
- Yandex Browser блокирует использование `ExtensionInstallForcelist`;
- стандартный Firefox требует подписанный XPI;
- установка через магазин или файл может требовать подтверждение пользователя.

YMus Desktop должен честно показывать пользователю используемый режим установки
и не обещать автоматизацию, которую браузер запрещает.

## 2. Поддерживаемая среда

### 2.1. Windows

MVP:

- Windows 10 22H2 x64;
- Windows 11 x64;
- интерфейс на русском языке;
- установка для текущего пользователя;
- административные права запрашиваются только для policy-режима.

После MVP:

- Windows 11 ARM64;
- английская локализация;
- portable-версия;
- установка для всех пользователей компьютера.

### 2.2. Браузеры MVP

| Браузер | Обнаружение | Рекомендуемый режим | Дополнительный режим |
|---|---:|---|---|
| Yandex Browser | Да | магазин или guided install | распакованное расширение |
| Google Chrome | Да | Chrome Web Store | policy для управляемого ПК |
| Microsoft Edge | Да | Edge Add-ons | policy для управляемого ПК |
| Mozilla Firefox | Да | подписанный XPI | `ExtensionSettings` policy |

После MVP:

- Opera;
- Opera GX;
- Brave;
- Vivaldi;
- Chromium;
- Firefox Developer Edition и ESR.

Каждый дополнительный браузер реализуется отдельным адаптером и проходит
отдельное тестирование. Нельзя считать все Chromium-браузеры одинаковыми:
политики и ограничения распространения отличаются.

## 3. Режимы установки расширения

### 3.1. Store-assisted

Основной режим для обычного пользователя.

1. Программа проверяет актуальную версию.
2. Открывает официальную страницу расширения в выбранном браузере.
3. Показывает короткий статус «Подтвердите установку в браузере».
4. После подтверждения повторно проверяет статус.

Плюсы:

- соответствует требованиям браузеров;
- автоматические обновления выполняет сам браузер;
- не требует административных прав.

### 3.2. Guided unpacked install

Режим для Yandex Browser и тестовых сборок Chromium. Для YMus это один из
ключевых режимов, потому что CRX-пакет не подходит: браузеры и сторонние
каталоги могут блокировать локальный CRX, а проекту нужна защита от подмены
расширения злоумышленниками.

1. Программа скачивает ZIP.
2. Проверяет подпись и SHA-256.
3. Распаковывает в постоянную директорию:
   `%LOCALAPPDATA%\YMus\extensions\chromium\current`.
4. Открывает страницу управления расширениями выбранного браузера.
5. Показывает пошаговый экран:
   - включить режим разработчика;
   - выбрать «Загрузить распакованное расширение»;
   - выбрать подготовленную папку.
6. После обновления программа атомарно заменяет содержимое `current`.
7. Пользователю предлагается перезапустить браузер или нажать «Обновить» на
   странице расширений.

Требование: путь `current` не должен меняться между версиями.

### 3.2.1. Максимально автоматизированный unpacked-flow

Цель продукта - сделать установку распакованного расширения максимально близкой
к «одна кнопка», но без хрупкого обхода защиты браузера.

Программа должна автоматически:

1. скачать последнюю версию расширения;
2. проверить подпись release manifest;
3. проверить SHA-256 архива;
4. распаковать пакет в постоянный путь;
5. закрыть старую локальную версию только после успешной проверки новой;
6. открыть страницу расширений выбранного браузера;
7. скопировать путь к папке `current` в буфер обмена;
8. показать поверх приложения короткую инструкцию с одним активным шагом;
9. после возврата в приложение проверить, появился ли YMus в браузере.

Программа может пытаться автоматизировать действия через официальные механизмы
операционной системы только если это не нарушает правила браузера и требует
явного согласия пользователя. В MVP автоматическое включение developer mode и
нажатие «Загрузить распакованное расширение» не считаются надежной частью
продукта, потому что UI браузеров меняется и эти действия специально требуют
пользовательского подтверждения.

Запрещается делать установку через:

- скрытую правку профиля браузера;
- имитацию кликов в браузере без видимого согласия пользователя;
- подмену внутренних файлов расширений;
- запуск браузера с небезопасными флагами как основной пользовательский режим.

Допускается отдельный режим «Лабораторная автоматизация» только для внутреннего
тестирования. Он не входит в публичный MVP и не должен быть доступен обычному
пользователю.

### 3.3. Managed policy install

Дополнительный режим для управляемых компьютеров и опытных пользователей.

- включается отдельным переключателем;
- перед применением показывает предупреждение;
- запрашивает UAC только после явного подтверждения;
- сохраняет резервную копию изменяемых policy-значений;
- позволяет удалить только те значения, которые создал YMus Desktop.

Особенности:

- Chrome вне Chrome Web Store может требовать домен, Azure AD или Chrome
  Enterprise Core;
- Edge вне Edge Add-ons ограничивает force-install на неуправляемых Windows;
- Firefox поддерживает `ExtensionSettings` с `install_url`;
- Yandex Browser не должен получать `ExtensionInstallForcelist`, поскольку
  браузер запрещает этот механизм.

### 3.4. Недопустимые способы

- изменение файлов профиля браузера при запущенном браузере;
- скрытое включение developer mode без действия пользователя;
- обход диалогов подтверждения;
- подмена файлов расширений, установленных из магазина;
- запуск браузера с небезопасными флагами без явного согласия;
- применение корпоративных политик без объяснения и подтверждения.

## 4. Пользовательские сценарии

### 4.1. Первый запуск

1. Приложение запускается.
2. Проверяет собственное обновление.
3. Загружает release manifest расширения.
4. Сканирует установленные браузеры.
5. Показывает главный экран со списком браузеров.
6. Пользователь отмечает браузеры.
7. Нажимает «Установить YMus».
8. Приложение выполняет доступные автоматические действия.
9. Для оставшихся шагов показывает инструкции по одному браузеру за раз.
10. Показывает итоговый отчет.

### 4.2. Обновление расширения

1. Приложение сравнивает установленную и последнюю версии.
2. Для store-версии предлагает открыть страницу расширения или проверить
   обновления в браузере.
3. Для unpacked-версии скачивает и атомарно заменяет файлы.
4. Для policy-версии проверяет URL и policy-конфигурацию.
5. Показывает результат для каждого браузера отдельно.

### 4.3. Удаление

Приложение должно:

- удалить локально скачанные пакеты;
- удалить созданные программой policy-значения;
- открыть страницу расширений для пользовательского удаления магазинной версии;
- не удалять чужие policy-значения или пользовательские данные браузера.

## 5. Интерфейс

### 5.1. Визуальный стиль

Интерфейс должен быть утилитарным, аккуратным и узнаваемым, без шаблонного
«нейросетевого» дизайна.

Основная палитра:

- фон: `#101010`;
- вторичный фон: `#181818`;
- границы: `#303030`;
- основной желтый: `#FFD400`;
- желтый при наведении: `#FFE24D`;
- основной текст: `#F5F5F5`;
- вторичный текст: `#A8A8A8`;
- ошибка: `#E5484D`;
- успех: `#46A758`.

Требования:

- без градиентных сфер, размытого фона и декоративных карточек;
- радиус элементов не более 8 px;
- желтый используется для действий и статусов, а не заливает весь интерфейс;
- основной шрифт интерфейса: Kanit;
- Kanit поставляется вместе с приложением локально, чтобы интерфейс не зависел
  от наличия шрифта в системе или подключения к сети;
- fallback-шрифты: Segoe UI, Arial, sans-serif;
- четкие иконки Lucide;
- анимации 120-180 мс, без декоративных длительных эффектов;
- доступность при масштабе Windows 125%, 150% и 200%.

### 5.2. Главный экран

Верхняя панель:

- логотип и название YMus;
- версия приложения;
- кнопка настроек;
- индикатор доступности сервера обновлений.

Основная область:

- версия последнего расширения;
- дата публикации;
- список найденных браузеров;
- чекбокс выбора для каждого браузера;
- иконка браузера;
- установленная версия YMus;
- последняя версия YMus;
- способ установки;
- статус: не установлено, актуально, требуется обновление, требует действия,
  ошибка.

Нижняя панель:

- основная кнопка «Установить» или «Обновить»;
- кнопка «Удалить» только после выбора установленного расширения;
- ссылка на журнал диагностики.

### 5.3. Дополнительные экраны

- пошаговая установка с прогрессом;
- подтверждение административных действий;
- настройки канала обновлений;
- диагностика;
- информация о приложении;
- экран результата;
- экран guided-установки распакованного расширения с копированием пути,
  открытием страницы расширений и проверкой статуса после действия пользователя.

### 5.4. Настройки

- канал обновлений: stable, beta;
- проверять обновления при запуске;
- автоматически скачивать обновление расширения;
- запускать приложение вместе с Windows - выключено по умолчанию;
- хранить предыдущую версию для отката;
- включить подробные журналы;
- сбросить локальный кэш.

Телеметрия по умолчанию выключена. В MVP ее можно не реализовывать.

## 6. Архитектура приложения

### 6.1. Рекомендуемый стек

**Desktop framework:** Tauri 2  
**Системная часть:** Rust  
**Frontend:** React, TypeScript, Vite  
**Стили:** CSS Modules или обычный CSS с design tokens  
**Менеджер пакетов:** npm  
**CI/CD:** GitHub Actions  
**Инсталлятор Windows:** NSIS для MVP, MSI как дополнительный артефакт

Почему Tauri:

- единая основа для Windows и будущей macOS-версии;
- небольшой размер приложения;
- системные операции реализуются в Rust;
- встроенная поддержка updater и подписанных обновлений;
- UI проще развивать на TypeScript/React.

### 6.2. Модули

```text
YMus Desktop
├── UI
│   ├── Dashboard
│   ├── Browser selection
│   ├── Installation wizard
│   ├── Settings
│   └── Diagnostics
├── Application services
│   ├── Update service
│   ├── Extension release service
│   ├── Installation coordinator
│   └── Settings service
├── Browser adapters
│   ├── Chrome adapter
│   ├── Edge adapter
│   ├── Yandex adapter
│   └── Firefox adapter
└── Platform services
    ├── Browser discovery
    ├── Registry and policy service
    ├── Download and verification
    ├── Atomic filesystem operations
    ├── Process detection
    └── Logging
```

### 6.3. Browser adapter

Каждый адаптер реализует общий контракт:

```text
detect() -> BrowserInstallation[]
getCapabilities(installation) -> InstallCapabilities
detectExtensionStatus(installation) -> ExtensionStatus
prepareInstall(release, mode) -> InstallPlan
executeStep(step) -> StepResult
removeManagedConfiguration() -> Result
openExtensionsPage() -> Result
```

Адаптер не должен самостоятельно скачивать файлы или изменять UI.

## 8. Обнаружение браузеров

Источники обнаружения:

- App Paths в Windows Registry;
- uninstall-записи HKCU и HKLM;
- стандартные директории `%LOCALAPPDATA%` и `%PROGRAMFILES%`;
- известные пути браузеров;
- версия EXE через Windows file version API.

Программа должна:

- находить несколько установок одного браузера;
- отличать stable, beta, dev и portable, если это возможно;
- не считать существование папки достаточным доказательством установки;
- проверять существование и подпись EXE;
- показывать пользователю путь в диагностике.

Профили браузера в MVP обнаруживаются только для отображения статуса.
Прямое редактирование файлов профиля запрещено.

## 9. Сервер обновлений

### 9.1. Компоненты

- HTTPS endpoint release manifest;
- объектное хранилище пакетов;
- CDN;
- опциональный endpoint выдачи короткоживущих ссылок на приватные артефакты;
- отдельные stable и beta каналы;
- неизменяемые артефакты релизов;
- release notes;
- CI-публикация.

Подойдет:

- Cloudflare R2 + CDN;
- Amazon S3 + CloudFront;
- совместимое S3-хранилище;
- небольшой API только при необходимости авторизации или rollout.

Статический CDN предпочтительнее собственного сложного backend на первом этапе.
Если ссылки на артефакты нельзя делать публичными, используется ticket-based
download API с коротким TTL ссылок. При этом секреты доступа к хранилищу
остаются только на сервере.

### 9.2. Release manifest

Пример находится в
[`release-manifest.example.json`](release-manifest.example.json).

Обязательные поля:

- версия схемы;
- канал;
- версия расширения;
- дата публикации;
- минимальная версия приложения;
- URL артефактов;
- размер;
- SHA-256;
- ссылки на магазины;
- Ed25519-подпись manifest.

### 9.3. Пакеты расширения

- Chromium unpacked ZIP;
- Chrome Web Store listing;
- Edge Add-ons listing;
- подписанный Firefox XPI;
- при необходимости отдельная сборка Firefox.

ID расширения для каждого магазина должен быть стабильным между версиями.

## 10. Безопасность

Обязательные требования:

- только HTTPS;
- release manifest подписывается Ed25519;
- публичный ключ вшит в приложение;
- каждый пакет проверяется по SHA-256;
- подпись проверяется до распаковки;
- распаковка защищена от Zip Slip;
- обновление файлов атомарное;
- предыдущая рабочая версия сохраняется для отката;
- скачанные исполняемые файлы не запускаются;
- shell-команды ограничены allowlist;
- administrative mode запускается только после подтверждения;
- все изменения registry/policy журналируются;
- удаляются только значения, созданные YMus Desktop;
- приложение и инсталлятор подписываются сертификатом Windows Code Signing;
- секретные ключи хранятся только в CI secrets или HSM/Key Vault.

Приложение не должно получать приватные данные браузера, cookies, историю,
пароли или содержимое профиля.

### 10.1. Защита от reverse engineering

Нужно исходить из правила: все, что попало в клиентское приложение, может быть
извлечено reverse-инженером. Поэтому в приложение нельзя вшивать долгоживущие
секреты, приватные ключи, master-токены, пароли к серверу или прямой доступ к
админским API.

Цель защиты клиента:

- усложнить массовую подделку приложения;
- усложнить автоматическое вытаскивание внутренних URL;
- снизить риск подмены пакета расширения;
- не позволить использовать приложение как постоянный ключ к серверу.

Защита не должна опираться только на обфускацию. Обфускация допустима как
дополнительный слой, но не как основной механизм безопасности.

### 10.2. Секреты и ссылки на сервер

В клиенте разрешено хранить:

- публичный ключ проверки release manifest;
- публичный base URL update API, если он не дает доступа к приватным данным;
- идентификатор канала обновлений;
- публичные store URL.

В клиенте запрещено хранить:

- приватный ключ подписи manifest;
- токен записи в хранилище;
- постоянный bearer-token к API;
- прямые приватные ссылки на закрытые артефакты;
- учетные данные CDN, S3, R2 или backend.

Рекомендуемая модель:

1. Приложение обращается к публичному endpoint `GET /v1/releases/{channel}`.
2. Endpoint возвращает подписанный release manifest.
3. Manifest содержит версии, хэши, размеры и идентификаторы артефактов.
4. Для приватных артефактов приложение запрашивает `POST /v1/artifacts/ticket`.
5. Сервер возвращает короткоживущую pre-signed URL на 1-5 минут.
6. URL привязана к версии, артефакту и каналу.
7. После скачивания приложение проверяет SHA-256 и подпись manifest.

Если артефакты публичные, безопасность все равно обеспечивается подписью и
хэшем. Скрытие URL не считается защитой.

### 10.3. Подпись и доверенная цепочка

Минимальная цепочка доверия:

```text
YMus Desktop binary
└── embedded public key
    └── verifies release manifest signature
        └── contains artifact SHA-256
            └── verifies downloaded ZIP/XPI
```

Требования:

- приватный Ed25519-ключ подписи хранится вне репозитория;
- подпись release manifest выполняется в CI/CD;
- ключ можно ротировать через список доверенных публичных ключей;
- старые ключи имеют `validUntil`;
- приложение отвергает manifest с неизвестной схемой или просроченным ключом;
- downgrade-атаки блокируются сравнением semver и минимальной версии.

### 10.4. Защита API от злоупотребления

Backend должен предполагать, что клиент можно разобрать.

Меры:

- rate limiting по IP, fingerprint и каналу;
- request nonce для ticket-запросов;
- короткий TTL ссылок;
- audit log выдачи ссылок;
- блокировка подозрительных user-agent и частот запросов;
- отдельные read-only credentials у CDN;
- отсутствие endpoint, который выдает список всех приватных артефактов;
- сервер не доверяет версии приложения, присланной клиентом, без проверки
  совместимости и подписи manifest.

Для публичного MVP можно начать со статического CDN и подписанного manifest.
Приватные tickets добавляются, когда появится реальная угроза массового
парсинга или платная модель распространения.

### 10.5. Защита бинарника

Для Windows-сборки:

- обязательная подпись приложения code signing сертификатом;
- проверка собственной подписи при запуске;
- проверка целостности критичных embedded resources;
- запрет запуска с подмененной директорией данных, если это ломает trust chain;
- минимизация строк с внутренними endpoint в бинарнике;
- release build без debug symbols;
- symbol files хранятся отдельно в приватном хранилище;
- включить базовую обфускацию frontend bundle;
- Rust-часть собирать с оптимизацией release и strip symbols;
- crash dumps не должны содержать секреты и полные URL pre-signed ссылок.

Дополнительные меры после MVP:

- commercial packer/protector только после оценки false-positive антивирусов;
- certificate pinning для update API с планом ротации;
- anti-tamper проверка критичных файлов;
- watermark build-каналов;
- server-side anomaly detection.

Важно: слишком агрессивные packer/anti-debug решения часто вызывают ложные
срабатывания антивирусов и ухудшают доверие пользователя. Для YMus приоритет:
подпись, прозрачная цепочка доверия, проверяемые обновления и отсутствие
секретов в клиенте.

### 10.6. Защита расширения

Для распакованного расширения:

- хранить только проверенную текущую версию в `current`;
- новая версия распаковывается в `staging`;
- после проверки `staging` атомарно становится `current`;
- предыдущая версия хранится в `previous`;
- запрещать запуск/подключение расширения из временной директории;
- manifest расширения проверяется после распаковки;
- версия в manifest должна совпадать с release manifest;
- запрещены внешние remote scripts;
- CSP расширения должна оставаться строгой;
- приложение должно обнаруживать ручную модификацию файлов `current` по
  manifest-хэшу и предлагать восстановление.

## 11. Обновление самого приложения

YMus Desktop проверяет собственное обновление отдельно от обновления расширения.

Требования:

- использовать Tauri updater;
- проверять подпись обновления;
- показывать release notes;
- не устанавливать приложение автоматически без согласия пользователя;
- поддерживать обязательное обновление только при несовместимости серверной
  схемы или критической уязвимости;
- после обновления восстанавливать незавершенный install plan.

## 12. Журналирование и диагностика

Логи хранятся в `%LOCALAPPDATA%\YMus\logs`.

Уровни:

- info - обычные действия;
- warn - ограничение браузера или требуемое действие;
- error - операция завершилась неуспешно;
- debug - включается пользователем.

Логи не должны содержать:

- cookies;
- токены;
- историю;
- полные данные профиля;
- персональные файлы пользователя.

Диагностический экран показывает:

- версию приложения;
- версию release manifest;
- найденные браузеры и пути;
- выбранный режим установки;
- результаты проверки подписи;
- последние ошибки;
- кнопку экспорта обезличенного отчета.

## 13. Ошибки

Минимальный набор кодов:

```text
NETWORK_UNAVAILABLE
UPDATE_SERVER_UNAVAILABLE
RELEASE_MANIFEST_INVALID
RELEASE_SIGNATURE_INVALID
ARTIFACT_HASH_MISMATCH
ARTIFACT_DOWNLOAD_FAILED
ARCHIVE_INVALID
BROWSER_NOT_FOUND
BROWSER_RUNNING
INSTALL_MODE_UNSUPPORTED
USER_CONFIRMATION_REQUIRED
ADMIN_PERMISSION_REQUIRED
POLICY_APPLY_FAILED
EXTENSION_VERIFICATION_FAILED
ROLLBACK_FAILED
```

Каждая ошибка содержит:

- стабильный код;
- понятное сообщение пользователю;
- технические детали для диагностики;
- рекомендуемое следующее действие.

## 14. Нефункциональные требования

- холодный запуск до главного экрана: не более 2 секунд на типичном ПК;
- приложение остается отзывчивым во время скачивания;
- скачивание отменяется пользователем;
- поддерживается прокси Windows;
- временные файлы очищаются после ошибки;
- операция повторного запуска идемпотентна;
- приложение не изменяет браузеры, которые пользователь не выбрал;
- все долгие операции показывают прогресс;
- UI работает без подключения к серверу и показывает кэшированный статус;
- crash-free rate целевой версии: не менее 99,5%.

## 19. Критерии готовности Windows MVP

MVP считается готовым, когда:

- приложение устанавливается подписанным NSIS-инсталлятором;
- UI использует локально поставляемый Kanit и корректные fallback-шрифты;
- находит Chrome, Edge, Yandex Browser и Firefox;
- получает и проверяет подписанный release manifest;
- скачивает актуальные пакеты;
- не устанавливает пакет при неверной подписи или SHA-256;
- не содержит приватных ключей, постоянных серверных токенов и секретов
  хранилища;
- не раскрывает pre-signed URL в логах, диагностике или crash dumps;
- позволяет выбрать несколько браузеров;
- корректно проводит пользователя через разрешенный способ установки;
- обновляет guided unpacked-версию без смены пути;
- не скачивает и не изменяет расширения без выбора пользователя;
- ведет диагностический журнал;
- умеет откатываться после неудачного обновления;
- успешно проходит тестовую матрицу Windows 10/11.

## 20. Решения, необходимые до начала реализации

1. Зарегистрировать постоянные extension ID.
2. Решить, публикуется ли расширение в Chrome Web Store, Edge Add-ons и AMO.
3. Выбрать домен и хранилище обновлений.
4. Создать ключ Ed25519 для release manifest.
5. Получить Windows Code Signing сертификат или настроить Azure Artifact
   Signing.
6. Определить, нужен ли managed mode в первой публичной версии.
7. Подготовить отдельную Firefox-совместимую сборку расширения.
